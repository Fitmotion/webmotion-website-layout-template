Quick Start

Last updated: Feb 22th, 2021

Introduction

Webmotion.cloud is a fully connected marketing plattform, specialized on several verticals like Fitness, Beauty, Ink, Art, Car, Med, Craft, Gastro and Immo. It builds highly adapted Websites and Landingpages for them and integrates them together with other Marketing Tools, like an annual marketing plan, a fully customized crm and many more. To achieve this, every customer must have the possibility, to exchange the content on the websites and landingpages on his own. Therefore we built a fully customized content management system, which builds on top of the website schema. For the best user experience, the content management system has to be perfectly adopted to each website and landingpage and should give the user the possibility to exchange exactly what he needs, but not more and not less. Therefore we need to tag each website, so the content management system knows, which content the user is allowed to change and which not.

The tag which is used for our webmotion.cloud:
<div data-cms-<tag-name>="<tag-content>">

Page Structure:

A page is always constructed with several sections

The section name should be tagged with:

<div data-cms-section="<section-name>">

Section Name

This section name must be unique unless it is exactly the same section and only the content differs.

Example:

Header
Section 1
Section 2
Section 3
...
Footer
 
<body> 
    <section data-cms-section="Header">Header</section>
    <section data-cms-section="Section 1">Section 1</section>
    <section data-cms-section="Section 2">Section 2</section>
    <section data-cms-section="Section 3">Section 3</section>
    <section >...</section>
    <section data-cms-section="Footer">Footer</section>
</body>

Section with content:

All html content elements should be marked with:

<div data-cms-element="<element-name>">

Element Name

The element must be unique within the scope of the parent. The scope of the parent can either be a section or a component.

For some simple text elements, add the following tag:

<div data-cms-element="<element-name>" data-cms-text="true">

For more complex elements, where an attribute has to be exchanged, add the following tag:

<div data-cms-element="<element-name>" data-cms-attributes="<attribute-name>:<attribute-type>">

You can also chain more attributes which should be exchangeable with a semicolon:

<div data-cms-element="<element-name>" data-cms-attributes="<attribute-name-1>:<attribute-type>;<attribute-name-2>:<attribute-type>">

Attribute Types

The following attribute types does exist:

  • IMG
  • VIDEO
  • STYLE_IMG
  • LINK
  • TEXT

Example:

Section 1

Lorem ipsum dolor sit amet, consectetuer adipiscing elit.
 
<section data-cms-section="Section 1">
    <h3 data-cms-element="Headline" data-cms-text="true">Section 1</h3>
    <div class="row">
        <div class="col-6" data-cms-element="Description" data-cms-text="true">Lorem ipsum dolor sit amet, consectetuer adipiscing elit.</div>
        <div class="col-6">
            <img src="..." alt="Image" data-cms-element="Image" data-cms-attributes="src:IMG">                              
        </div>
    </div>
</section>

Which content should be exchangeable?

As you can see in the above example, the alt Tag of the image is not tagged in the data-cms-attributes tag. Because the alt tag is not a direct content, which the end customer can see, it should therefore not be visible in the content management system.

Section with more complex content (e.g. Background Images):

Sometimes we want the user to exchange some more complex nested attributes, like a style attribute. This is often the case for background images. To do this, first of all, the background-image should be declared directly in the style tag and NOT in the .css file. Then we can use the following tag, to exchange the nested attribute:

<div data-cms-element="<element-name>" data-cms-nested-attributes="style=background:STYLE_IMG;">

The different between IMG and STYLE_IMG

As you can see, we used the STYLE_IMG attribute type in the example above. This is the case, because for a STYLE_IMG, we can't write the source directly into the attribute (e.g. background). Instead, we have to use the url() identifier.

We can also chain multiple nested attributes with a comma AND multiple attributes, which are nested with a semicolon: 

<div data-cms-element="<element-name>" data-cms-nested-attributes="style=background:STYLE_IMG,color:TEXT;my-custom-attribute=href:LINK;">

Section including components:

Components

A Component is defined as content in a section, which is duplicatable and removable. Sections can have multiple different components, from which the user then can select, to add to the section.

We first need to define a component wrapper, in which the components are rendered and the category. The category defines, which components are allowed to be rendered into this component list. 

<div data-cms-component-list="<component-list-name>" data-cms-component-category="<component-category>"></div>

Then we can tag the components in the component list. Each component, which is structured exactly the same, should be called the same. A component, which is structured a bit differently, must be called not the same. E.g. we have four components in a component list. Two of them just contain some text and two of them contain an image. Therefore we have to call both with the text the same (e.g. Component Text) and both with the image the same (e.g. Component Image). They still have all the same component category, because they are rendered into the same component list 


<div data-cms-component-list="<component-list-name>" data-cms-component-category="<component-category>">
    <div data-cms-component="<component-1>" data-cms-component-category="<component-category>"></div>
    <div data-cms-component="<component-2>" data-cms-component-category="<component-category>"></div>
    <div data-cms-component="<component-1>" data-cms-component-category="<component-category>"></div>
    <div data-cms-component="<component-2>" data-cms-component-category="<component-category>"></div>
</div>

As you can see, we can use the same Element Name (Headline) for each component and the section. Because every element is inside another scope, the element naming is still unique inside the scope.

Component Order

Because the customer can remove, duplicate and add components to a component list, the order of the components must not be defined through class names like order.

Component Structure

Components should not depend on each other. This is espacially important for the size of a component. In the above example, the size of the text component must not depend on the size of the image component. It is possible, that the user removes both image components and still want the components to look the same.

Folder structure & File Naming

Basic

In order to have a uniform folder structure, please use the structure and naming mentioned below.

  • src
    • partials
      • head.html
      • sections
      • popup-sections
      • components
    • custom-js
    • custom-style
      • style.css
    • assets
      • library
      • third-party
      • media
    • index.html
    • news.html
    • ...

Partials

The partials folder should include the html for all existing sections, popup-sections and components. In addition to that, it should include the basic head content with all imports, metadata and external scripts (e.g. google analytics, facebook pixel).

Example partials/sections:

header.html

<section data-cms-section="Header"> 
    ...
</section>
footer.html

<section data-cms-section="Footer"> 
    ...
</section>
services.html

<section data-cms-section="Services"> 
    ...
</section>

Example partials/popup-sections:

popup.html

<section data-cms-popup-section="Popup"> 
    ...
</section>

Example partials/components:

service.html

<div data-cms-component="Service"> 
    ...
</div>

Example partials/head.html:

The head.html should include all metatags, title, scripts and imports. The metatags and the title will be exchanged in the cloud, so it is ok to have a similar head for all pages. 


<meta charset="utf-8">
<title>Page Title</title>
...

Custom Javascript

Custom Javascript should be placed into the "custom-js" folder. Custom Javascript Files should have a self explaing naming, e.g. use the name "custom-form-validation.js" for a custom form validation script.

Custom Style

Custom stlye should be placed into the "custom-style" folder. Inside the folder there must be a file named "style.css", which includes all color vars.

Other styles can be placed into seperate ".css" files, or also inside the "style.css" file.

Color Vars in the "style.css" file:

:root {
    --color-1: #000000 !important;
    --color-2: #101010 !important;
    --color-3: #010101 !important;
    --color-4: #111111 !Important;
    ...
}

Assets (JS-Library, Third-Party, Media)

The asset folder should contain the already existing JS-Library, Third-Party Packages and Media (dummy-images).

Script Imports

Because we can't know how the customer builds his website, all scripts must be imported inside each page.

In order to don't forget a script import, always build the partials/head.html first and include this as last step into the actual pages.

Global Variables (Google Analytics, Facebook Pixel, ...)

it is possible to define global variables inside the html layout. This is espacially required for google analytics, facebook pixel and other external scripts.

In order to make these scripts working, the customers id has to be injected at the correct position.

The following variables exists (It is also possible to define custom variables if necessary):

  • --data-cms-var-facebook-pixel--
  • --data-cms-var-google-analytics--
  • --data-cms-var-usercentrics--

Tag:

--data-cms-var-<var-name>--

Usage Example:



<script type="text/plain" data-usercentrics="Google Analytics" async src="https://www.googletagmanager.com/gtag/js?id=--data-cms-var-google-analytics--"></script>
<script type="text/plain" data-usercentrics="Google Analytics">
    window.dataLayer = window.dataLayer || [];
function gtag() {
    dataLayer.push(arguments);
}
gtag('js', new Date());
gtag('config', '--data-cms-var-google-analytics--', {
    'anonymize_ip': true
});
</script>

Global Vars Usage:

As you can see in the above example, we used the global var one time in the "src" attribute of the script and one time in the script itself.

Global vars will replace the var exactly with the customers id, no matter where the global var is used.
