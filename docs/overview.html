Tagging Overview

Last updated: Feb 22th, 2021

Page Tagging

Pages must have a page name and a page type, in order to identify the Page in the website builder.

Pages can have one of the following page types:
  • HOMEPAGE
  • SERVICE
  • CONTACT
  • BOOKING
  • SALE
  • NEWS
  • CAREER
  • IMPRINT
  • PRIVACY_POLICY
  • TERMS_CONDITIONS
  • ABOUT_US
  • THANKS
  • OTHERS
The page type should always be deinfed on the head tag together with a page name.
Homepage example:

<head data-cms-page="Homepage" data-cms-page-type="HOMEPAGE"> 
    ...
</head>

Section Tagging

Basic

Sections must be tagged with a section name. This section name must be unique, unless sections have exactly the same schema. 


<div data-cms-section="<Section Name>"> 
    ...
</div>

Fixed

Some Sections should always stay at the same position. This means, the user can not move this section inside the page to another position. It is possible to set the first three sections as fixed, as well as the last three sections. In addition to that, a section can be set to independent, if it does not matter where the section is defined inside the html.

  • FIRST
  • SECOND
  • THIRD
  • THIRDLAST
  • SECONDLAST
  • LAST
  • INDEPENDENT

Tag:

data-cms-fixed

Example:


<div data-cms-section="<Section Name>" data-cms-fixed="FIRST"> 
    ...
</div>

Unique

Some Sections should always be unique on a page. This includes espacially Header and Footer, but can also include some other sections, which should not be displayed twice.

Tag:

data-cms-unique

Example:


<div data-cms-section="<Section Name>" data-cms-unique="true"> 
    ...
</div>

Global

Some Sections should be the same on every page. This mainly includes the Header and Footer, but could also be the case for other other sections like a map.

Tag:

data-cms-global

Example:


<div data-cms-section="<Section Name>" data-cms-global="true"> 
    ...
</div>

Internal

Some Sections are just for internal use and contain no content. Therefore the User should not see this section in the website builder.

Tag:

data-cms-internal

Example:


<div data-cms-section="<Section Name>" data-cms-internal="true"> 
    ...
</div>

Database Module

Some Sections should read their data from the database. This is for example the case for customer master-data (Company Name, Address, Logo), services and memberships. For a full list of database modules read this.

Tag:

data-cms-database-module

Example:


<div data-cms-section="<Section Name>" data-cms-database-module="<module-name>"> 
    ...
</div>

Database Category

It is also possible to further categorize the database module, which should be loaded. Click here, to see available modules.

Tag:

data-cms-database-category

Example:


<div data-cms-section="<Section Name>" data-cms-database-module="<module-name>" data-cms-database-category="<category-name>"> 
    ...
</div>

How does the injection of the database content work?

Database Content is injected by two different ways. Which one is used, depends on the data-cms-internal Tag.

Internal Tag is set

If the internal Tag is set, the database Content is injected once at the beginning, after creating the website. After that, it always injected if the database content changes. This is for example the case, if the customer changes his logo in the cloud or he adds a new membership. For database content, which exists multiple times (e.g. a customer has three memberships, which he wants on the website), the section will be copied as often, as database content exists. (E.g. the customer has three memberships and the membership is displayed in one section, this section will be copied three times and filled with the dedicated content.

Internal Tag is NOT set

If the internal Tag is NOT set, the database content will be injected, if the user actively selects it. This is for example the case, if the user adds a new service to his service page. He then can select from all predefined service templates (e.g. Sauna, Fitness-Circle, ...). If he selects one, the predefined data will be directly injected into the section / component.

Component List Tagging

Basic

Component Lists must be tagged with a component list name and a component category, to identify, which components are allowed to be rendered into this component list. 


<div data-cms-component-list="<Component List Name>" data-cms-component-category="<Component Category>"> 
    ...
</div>

Use of Component Lists:

Component Lists can only appear once in their parent scope. E.g. a section can only have one component list. If multiple component lists are required, they have to be wrapped into another component list and then again defined as components.

Example:

Section 1 having a Component List

Component 1, which is just a wrapper for another component list.
Component List 1 - Component 1
Component List 1 - Component 2
Component 2, which is just a wrapper for another component list.
Component List 2 - Component 1
Component List 2 - Component 2

<div data-cms-section="Section 1">
    <h3 data-cms-element="Headline" data-cms-text="true">Section 1 having a Component List</h3>
    <div data-cms-component-list="Outer Component List" data-cms-component-category="component-list-component">
        <div data-cms-component="Component List Wrapper" data-cms-component-category="component-list-component">
            <h5>Component 1, which is just a wrapper for another component list.</h5>
            <div data-cms-component-list="Inner Component List" data-cms-component-category="text-component">
                <div data-cms-component="Text Component" data-cms-component-category="text-component">Component List 1 - Component 1</div>
                <div data-cms-component="Text Component" data-cms-component-category="text-component">Component List 1 - Component 2</div>
            </div>
        </div>
        <div data-cms-component="Component List Wrapper" data-cms-component-category="component-list-component">
            <h5>Component 2, which is just a wrapper for another component list.</h5>
            <div data-cms-component-list="Inner Component List" data-cms-component-category="text-component">
                <div data-cms-component="Text Component" data-cms-component-category="text-component">Component List 2 - Component 1</div>
                <div data-cms-component="Text Component" data-cms-component-category="text-component">Component List 2 - Component 2</div>
            </div>
        </div>
    </div>
</div>

Component List and Component Naming:

As you may noticed, we named the inner component lists, components and categories all the same. This should be done, because the inner components have exactly the same schema.

Minimum number of components

A component list should often have at least one or two components inside. We can define the minimum number of components, which must be inside the component list.

Tag:

data-cms-min-components

Example:


<div data-cms-component-list="<Component List Name>" data-cms-min-components="1"> 
    ...
</div>

Maximum number of components

Sometimes we don't want the customer to add as many components to a component list as he wants. Therefore we can specify a maximum amount of components, which can be in this component list.

Tag:

data-cms-max-components

Example:


<div data-cms-component-list="<Component List Name>" data-cms-max-components="4"> 
    ...
</div>

Internal

Some component lists are just for internal use and contain no content or the content will be injected automatically. Therefore the User should not see this component list in the website builder.

Tag:

data-cms-internal

Example:


<div data-cms-component-list="<Component List Name>" data-cms-internal="true"> 
    ...
</div>

Database Module

Some Component Lists should read their data from the database. This is for example the case for services and memberships. For a full list of database modules read this.

Tag:

data-cms-database-module

Example:


<div data-cms-component-list="<Component List Name>" data-cms-database-module="<module-name>"> 
    ...
</div>

Database Category

It is also possible to further categorize the database module, which should be loaded. Click here, to see available modules.

Tag:

data-cms-database-category

Example:


<div data-cms-component-list="<Component List Name>" data-cms-database-module="<module-name>" data-cms-database-category="<category-name>"> 
    ...
</div>

How does the injection of the database content work?

Database Content is injected by two different ways. Which one is used, depends on the data-cms-internal Tag.

Internal Tag is set

If the internal Tag is set, the database Content is injected once at the beginning, after creating the website. After that, it is always injected if the database content changes. This is for example the case, if the customer changes his logo in the cloud or he adds a new membership. For database content, which exists multiple times (e.g. a customer has three memberships, which he wants on the website), the component list generates as many components, as content exists (e.g. three components for three memberships). The component type can be specified using the data-cms-database-category Tag at Component Level.

Internal Tag is NOT set

If the internal Tag is NOT set, the database content will be injected, if the user actively selects it. This is for example the case, if the user adds a new service to his service page. He then can select from all predefined service templates (e.g. Sauna, Fitness-Circle, ...). If he selects one, the predefined data will be directly injected into the component.

Component Tagging

Basic

Components must be tagged with a component name. This component name must be unique, unless components have exactly the same schema. They also require a component category, to identify in which component list they can be rendered. 


<div data-cms-component="<Component Name>" data-cms-component-category="<component-category>"> 
    ...
</div>

Unique

Some Components should always be unique in a component list.

Tag:

data-cms-unique

Example:


<div data-cms-component="<Component Name>" data-cms-component-category="<component-category>" data-cms-unique="true"> 
    ...
</div>

Element Tagging

Basic

Elements can appear in the scope of sections or in the scope of components. They must have a unique element name inside the scope. 


<div data-cms-element="<Element Name>"> 
    ...
</div>

Text

The simplest exchangeable content, which can be defined on an element, is Text. If this tag is set, the customer can exchange the text content in this element on the website builder.

Tag:

data-cms-text

Example:


<div data-cms-element="Headline" data-cms-text="true"> 
    ...
</div>

Element Text

The text content of an element should not include other html elements. Instead it should only contain pure text. However it is possible in the future to add some basic html text tags inside the text content like head or paragraph elements. It is NOT possible to add html tags containing some custom classes inside the text. The customer in the website builder can not add this custom tag elements, so they break the design.

Good Example:

<div data-cms-element="Headline" data-cms-text="true"> 
Lorem ipsum dolor sit amet, consectetuer adipiscing elit.
</div>
Bad Example (Don't do this):

<div data-cms-element="Headline" data-cms-text="true"> 
Lorem ipsum dolor sit amet, <span class="custom-strong-class">consectetuer</span> adipiscing elit.
</div>

Text Margin

It is really important not to use spaces defined with "&nbsp;" as margin for a text element. The customer will not be able to add this margin, the same it is required in the desing.

Always use a css margin on the element instead (e.g. the bootstrap class "mr-2").

Attributes

It is also possible to exchange element attributes, which are mainly images, videos, or links. Each attribute which is exchangeable needs an attribute type, which defines what this attribute contains.

Exsiting Attribute Types:

  • IMG
  • VIDEO
  • STYLE_IMG
  • LINK
  • TEXT

Schema:

The attributes tag has the following schema: <attribute-1>:<attribute-type>;<attribute-2>:<attribute-type>;

Attribute Type Link

The Link Attribute Type can include mutliple different links, like an internal link to another page, an external link, tel links, mailto links, or even just a link for opening a popup.

Tag:

data-cms-attributes

Example:


<img data-cms-element="Image" src="path-to-image" data-cms-attributes="src:IMG">

Nested Attributes (e.g. Background Image in style)

It is also possible to exchange element nested attributes, which are mainly background images. Each nested attribute which is exchangeable needs an attribute type, which defines what this attribute contains.

Exsiting Attribute Types:

  • IMG
  • VIDEO
  • STYLE_IMG
  • LINK
  • TEXT

Schema:

The nested attributes tag has the following schema: <parent-attribute-1>=<nested-attribute-1>:<attribute-type>,<nested-attribute-2>:<attribute-type>;<parent-attribute-2>=<nested-attribute-1>:<attribute-type>,<nested-attribute-2>:<attribute-type>;

Tag:

data-cms-nested-attributes

Example:


<div data-cms-element="Background Image" style="background-image: url(path-to-image.png);" data-cms-nested-attributes="style=background-image:STLYE_IMG">

Attribute Type IMG vs. STLYE_IMG

IMG:

At the attribute type IMG, the source to the image is directly injected into the attribute.

Example:

The Layout Schema is defined like the following: 


<img data-cms-element="Image" src="..." data-cms-attributes="src:IMG">

If the customer now wants to inject an image with the path "my/path/to/the/image.png", this path is directly injected into the defined "src" attribute and results in the following html element: 


<img src="my/path/to/the/image.png">

STYLE_IMG:

At the attribute type STYLE_IMG, the source to the image is injected into the attribute with an enclosing "url()".

Example:

The Layout Schema is defined like the following: 


<div data-cms-element="Background Image" style="background-image: url(...);" data-cms-nested-attributes="style=background-image:STYLE_IMG">

If the customer now wants to inject an image with the path "my/path/to/the/image.png", this path is injected into the defined nested attribute "style" at "background-image" with an enclosing "url()" and results in the following html element: 


<div style="background-image: url(my/path/to/the/image.png);">

Internal

Some attributes should not be displayed to the customer. Instead the content should be injected from database data. Therefore the data-cms-internal Tag can be used together with the data-cms-database-module Tag.

Tag:

data-cms-internal

Example:


<div data-cms-element="Address" data-cms-text="true" data-cms-internal="true" data-cms-database-module="master-data">
Customer Address
</div>

Database Module

Some Elements should read their data from the database. This is for example the case for customer master-data (Company Name, Address, Logo). For a full list of database modules read this.

Tag:

data-cms-database-module

Example:


<div data-cms-element="Address" data-cms-text="true" data-cms-internal="true" data-cms-database-module="master-data">
Customer Address
</div>

Element Database Data and the internal Tag

On element scope, the data-cms-database-module Tag can only be used together with the internal Tag. This results in an element, where the data is always only injected from the database. The customer does not see this element.

Using the Database Module Tag inside of Database Sections/Components

It is also possible to use the database module Tag for a single element, inside of a section or component, where the data should also be injected from the database.

Use Case:

This could be the case for the footer, where most of the data is injected from the "master-data" Database-Module (e.g. address, company name, opening hours, ...). But the footer should maybe also show a map pointing to the correct address. Therefore we need to use the "map-data" Database-Module, to get the exact coordinates for the customer address.

Example:


<section data-cms-section="Footer" data-cms-database-module="master-data"> 
    <div data-cms-element="address" data-cms-text="true"></div> 
    <div data-cms-element="company-name" data-cms-text="true"></div>
    <div id="map">
        <div data-cms-element="map-coordinate-long" data-cms-text="true" data-cms-internal="true" data-cms-database-module="map-data"></div>
        <div data-cms-element="map-coordinate-lat" data-cms-text="true" data-cms-internal="true" data-cms-database-module="map-data"></div>
    </div>                                                ...
</section>

In this example, the address and the company-name are both injected from the parent scope database module "master-data", because the "master-data" module contains the keys "address" and "company-name" (see here).

The "map-coordinate-long" and "map-coordinate-lat" elements data instead, will be injected from the other database module "map-data", because they contain a direct database module tag together with an internal tag.

Database Category

It is also possible to further categorize the database module, which should be loaded. Click here, to see available modules.

Tag:

data-cms-database-category

Example:


<div data-cms-element="Address" data-cms-text="true" data-cms-internal="true" data-cms-database-module="master-data" data-cms-database-category="<database-category>">
Customer Address
</div>

Filter (Images and Videos)

Sometimes we want the user to inject only some custom images or videos into an Image or Video Element. This is espacially the case for transparent images. To achieve this, the filter tag can be used, together with an image/video type.

Exsiting Filter Tags:

  • transparent

Tag:

data-cms-filter

Example:


<img data-cms-element="Image" src="path-to-image" data-cms-attributes="src:IMG" data-cms-filter="transparent">

Image Width and Height (Mandatory)

In order to make a responsive website layout, it is mostly mandatory to use images, which have the correct width and height. We need to give the customer in the website builder the possibility to crop the image he wants to inject, to fit perfectly in the layout.

Therefore the website builder needs to know the exact width and height of the image, which should be injected. In order to achieve this, we need to tag each image with the corresponding tags.

Tags:

data-cms-image-width

data-cms-image-height

Example:


<img data-cms-element="Image" src="path-to-image" data-cms-attributes="src:IMG" data-cms-image-width="1920" data-cms-image-height="1080">

Width and Height

It is really important, to set the correct tags for each image in the website layout. Otherwise the website-buider can't give the customer the correct cropping size, which results in a breaking design.

Tenant data only

For some website content, espacially images and videos, we don't want the customer to inject his own images or videos, because it is very likely, that they will break the design. In this case, we can use the tenant data only tag, which only allows the customer to choose from the tenant images in the website builder.

Tag:

data-cms-tenant-data-only

Example:


<img data-cms-element="Image" src="..." data-cms-attributes="src:IMG" data-cms-tenant-data-only="true">

Info Box

For some elements, the customer should get some more information. This is for example the case for "h1" and "h2" tags. These tags will be used for SEO meta tagging and should therefore be considered wisely.

Exsiting Info Box Types:

  • h1
  • h2

Tag:

data-cms-info-box

Example:


<h1 data-cms-element="Headline" data-cms-text="true" data-cms-info-box="h1">
My Headline
</h1>

Navigation Tagging

Basic

The website navigation has to be tagged seperately and has it's own schema. Each website can have exactly one main navigation (Other navigations has to be handled via Component Lists).

The Navigation Schema supports as many sub-items, as required, however more than two sub-items may be sufficient. 


<div data-cms-navigation="main-navigation"> 
    ...
</div>

Nav Items

A Navigation Item is always tagged with the corresponding nav-item tag. It doesn't matter on which level the nav-item is, it requires always the same tag. Inside the tag, the nav-level is defined. As already mentioned, the nav-level can be as deep as required, but mostly it should be in the range from 1 to 3.

In addition to the actual nav-item, this tag has also to be used as container for sub-items. In this case, it has to be defined as data-cms-nav-item="<parent-level>_<children-level>".

Example:

For the container on nav-level 1, which holds all sub-items on nav-level 2, it should be defined as data-cms-nav-item="1_2".

Nav Item Active Class

Together with the nav-item tag, we always have to define an tag, which holds the required active class, which should be injected into the nav-item class list, if this page is active. This class should mostly called "active", but can differ on sub-nav-items, why we have to define it on each nav item.

How does the injection of the active class works?

The nav item active class is injected into the nav-item class list, if the nav-item link equals the current page.

Example:

We have two pages: A homepage with the filename index.html and a news page with the filename news.html.

The Navigation will be injected into both pages and looks like the following: 


<div data-cms-navigation="main-navigation"> 
    <div data-cms-nav-item="1" data-cms-active-class="active">
        <a href="index.html" data-cms-nav-item-text="true" data-cms-nav-item-link="href">Homepage</a>
    </div>
    <div data-cms-nav-item="1" data-cms-active-class="active">
        <a href="news.html" data-cms-nav-item-text="true" data-cms-nav-item-link="href">News</a>
    </div>
</div>

The Navigation Injector compares the current filename with the link of the nav-item. If the filename equals the link in the nav-item, the active class will be injected into the class list.

Result:


<!--index.html-->                                                     
<div data-cms-navigation="main-navigation"> 
    <div class="active">
        <a href="index.html">Homepage</a>
    </div>
    <div>
        <a href="news.html">News</a>
    </div>
</div>

<!--news.html-->                                                     
<div data-cms-navigation="main-navigation"> 
    <div>
        <a href="index.html">Homepage</a>
    </div>
    <div class="active">
        <a href="news.html">News</a>
    </div>
</div>

data-cms-active-class

    <div data-cms-nav-item="1" data-cms-active-class="active">

    </div>
    <div data-cms-nav-item="1" data-cms-active-class="active">
        ...
    </div>
    <div data-cms-nav-item="1" data-cms-active-class="active">
        ...
        <div data-cms-nav-item="1_2">
            <div data-cms-nav-item="2" data-cms-active-class="sub-active">
                ... 
            </div>
            <div data-cms-nav-item="2" data-cms-active-class="sub-active">
                ...          
            </div>                                
        </div>                                  
    </div>
</div>

Sub nav-item active class

As you can see in the above example, we can use different active classes for different nav-levels.

</div>

Together with the nav-item text, the child element holds also the nav-item link, in order to make the navigation functional. Because the link is not always defined with the basic "href" attribute, it has to be defined with this additional tag (This is for example the case for popup-section links).

        <div data-cms-nav-item-text="true">Services</div>

            </div>
            <div data-cms-nav-item="2" data-cms-active-class="sub-active">
                <a href="index.html" data-cms-nav-item-text="true" data-cms-nav-item-link="href">Service Category 1</a>
            </div>                                
        </div>                                  
    </div>
</div>

Nav Item holding sub nav-items

As you can see in the above example, the "Services" Nav Item does not have a nav-item-link tag itself, because it acts only as parent nav-item for some sub nav-items.

Form Tagging

Basic

All forms will be sent to the CRM of the cloud. Forms should always send a name and a mail to the cloud. All other fields are optional. Forms are mostly simple contact forms, including name, mail, number and a customer message. However, forms can also be more complex and even used for online sales and bookings.

All forms should have the form tag with a corresponding name.

Tag:

data-cms-form

Example:


<form data-cms-form="Contact Form"> 
    ...
</form>

Input Fields

A form can have multiple different input/select fields. The name of this field must equal exactly the name defined below. Otherwise, the data will not be evaluated correctly in the cloud.

Input Fields Personal Information:

  • title: e.g. Phd., Prof.
  • gender: e.g. M, W
  • mail: e.g. Max@Mustermann.de
  • prename: e.g. Max
  • surname: e.g. Mustermann
  • birthday: e.g. 01.10.1990
  • address: e.g. Hauptstraße
  • houseNumber: e.g. 21
  • zipCode: e.g. 34567
  • city: e.g. Bruchsal
  • phoneNumber: e.g. +49123456789

Input Fields Bank Account Information:

  • accountHolder: e.g. Max Mustermann
  • iban: e.g. DE11 0000 1111 0000 1111 0000 11
  • bic: e.g. XXXXXXXXXX
  • bankName: e.g. Sparkasse

Input Fields Contact Form:

  • customerMessage: e.g. Please contact me for a trial training.

Input Fields Metadata (Hidden):

  • type: e.g. Lead / Member / Question
  • tags: e.g. new / complaint / booking / sale

Input Fields Membership (Hidden):

  • membershipId: Will be injected automatically, but is still required for membership forms

Basic Contact Form


<form data-cms-form="Contact Form"> 
    <input type="text" autocomplete="given-name" name="prename" data-cms-required>
    <input type="text" autocomplete="family-name" name="surname">
    <input type="email" autocomplete="email" name="mail" data-cms-required>
    <input type="tel" autocomplete="tel" name="number">
    <textarea name="customerMessage" data-cms-required></textarea>
</form>

Form Validation and Form Submit

Form Validation

As you may noticed in the above example, the prename and mail input have both an additional tag data-cms-required. This tag can be used, together with the library form-validation for built-in form-validation.

It is still possible to make a custom form-validation (even with the library functionality). This is often required for a step based input form.

Generally, the form-validation is not injected automatically. If the implemented form needs a form-validation, you have to import at least the basic form-validation script from the library.

Form Submit

The form submition is injected automatically, because the form validation differs in the domain, where we have to send the data adapted to the customer.

Do not overwrite the form submit!

As mentioned above, the form submit is injected automatically by the cloud. In order to have a working form submit, it is not allowed to overwrite the form submit for other functionality like a form validation or others.

If you want to implement a custom from validation, you can do this also by the event handler 'click' on the submit button and stop the propagation of the event. In this case the submit function will not be called and the form-validation still works as expected.

Popup Section Tagging

Basic

Additional to basic sections, it is also possible to tag popup sections. Popup Sections have basically the same schema as normal sections, with the difference, that they should only be visible after a click on some link. They should also be visible as overlay above the normal sections. For this type of sections you can use the popup section tag.

Possible additional Tags:

  • Unique
  • Global
  • Internal
  • Database Modules

Tag:

data-cms-popup-section

Example:


<section data-cms-popup-section="<Popup Section Name>"> 
    ...
</section>

Popup Section Open

The customer should have the possibility, to open a specific popup section with a custom link. The popup open functionality should therefore be completely generic.

In order to achieve this, don't implement the open functionality on your own. Instead, use the popup section controller from the library. If you need additional functionality (e.g. a fade in), you can use the in-built listeners of the controller, also described in the library.

Tag:

data-cms-popup-open

Example:

In the example below, a click on the element "Popup Open Link", which is in the scope of the section "Section 1" will display the popup section "My Popup Section". 


<body>                                            
    <section data-cms-popup-section="My Popup Section"> 
        ...
    </section>
    <section data-cms-section="Section 1">
        <div data-cms-element="Popup Open Link" data-cms-popup-open="My Popup Section" data-cms-attributes="data-cms-popup-open:LINK;">
    </section>
</body>

Popup Section Open

As you may noticed in the above example, we defined the data-cms-popup-open in the data-cms-attributes Tag as LINK. By doing so, the customer can change this link in the website builder and also change the link to an internal or external page link.

Popup Section Open functionality

The internal implementation of the popup sections works with the basic style display attribute. All popup-sections should initially be set to style.display = "none". A click on data-cms-popup-open including the corresponding Popup Section name, will result in setting the popup section to style.display = "block".

Popup Section Close

For a best user experience, an open popup section should also display an additional close Button, which can be used to close all sections which are currently open.

In order to achieve this, don't implement the close functionality on your own. Instead, use the popup section controller from the library. If you need additional functionality (e.g. a fade out), you can use the in-built listeners of the controller, also described in the library.

Tag:

data-cms-popup-close

Example:

In the example below, a click on the first div will close all currently opened popup sections. The popup section close div will be displayed automatically, if a popup section is shown. 


<body>
    <div data-cms-popup-close="true"> 
        ...
    </div>                                            
    <section data-cms-popup-section="My Popup Section"> 
        ...
    </section>
    <section data-cms-section="Section 1">
        <div data-cms-element="Popup Open Link" data-cms-popup-open="My Popup Section" data-cms-attributes="data-cms-popup-open:LINK;">
    </section>
</body>

Popup Section Close

As you may noticed in the above example, the popup close div is in the scope of the <body> Tag. This is just one possible implementation. The close button can also be in the scope of the popup section itself.

A click on the close button will always close all popup sections. If you reference a popup section from another popup section, there should always be a back button in addtion to the close button.

Popup Section Back

For the best user experience, you should always add a back button for popup sections in the website layout. Even if the Layout does not include stacked popup-sections on its own, it could still be possible, that the customer implements stacked popup-sections on his own.

In order to achieve this, don't implement the back functionality on your own. Instead, use the popup section controller from the library. If you need additional functionality (e.g. a fade out), you can use the in-built listeners of the controller, also described in the library.

Tag:

data-cms-popup-back

Example:

In the example below, a click on the element "Popup Open Link 1" in the scope of "Section 1", will open the Popup Section with the name "My Popup Section 1". After this popup section is opened, the user can click on the element "Popup Open Link 2" in order to open the popup section "My Popup Section 2".

This results in a displayed back button, which the user know can click, to jump back to the popup section "My Popup Section 1" again. 


<body>
    <div data-cms-popup-back="true"> 
        ...
    </div>   
    <div data-cms-popup-close="true"> 
        ...
    </div>                                            
    <section data-cms-popup-section="My Popup Section 1"> 
        <div data-cms-element="Popup Open Link 2" data-cms-popup-open="My Popup Section 2" data-cms-attributes="data-cms-popup-open:LINK;">
    </section>
    <section data-cms-popup-section="My Popup Section 2"> 
        ...
    </section>
    <section data-cms-section="Section 1">
        <div data-cms-element="Popup Open Link 1" data-cms-popup-open="My Popup Section 1" data-cms-attributes="data-cms-popup-open:LINK;">
    </section>
</body>

Popup Section Back

As you may noticed in the above example, the popup back div is in the scope of the <body> Tag. This should be mostly the expected case. Otherwise the back button may not be shown.

Copies Tagging

Basic

As you may noticed in the previous sections, it is not possible to define an element or a component list two times, inside the parent scope, because it is not allowed, to name two elements the same.

Somtimes however, it may be the case that we want the exact same content inside two elements or wen want the content of a component list also inside another component list.

This could be the case for a step based membership form for example. We present some data on each step and we want to display a conclusion with all the previous data again on the last step.

In order to make this possible, we have to define the copy tag, with the corresponding element or component list name.

Tags:

data-cms-copy-element

data-cms-copy-component-list

Example:


<section data-cms-section="<Section 1>">
    <!--step 1-->
    <div data-cms-element="Membership Price"></div>
    <div data-cms-component-list="Membership Services">
        ...
    </div>
    <!--step 2-->
    <div data-cms-copy-element="Membership Price"></div>
    <div data-cms-copy-component-list="Membership Services">
        ...
    </div>
</section>

Copy Tags

As you can see in the above example, we defined the element and component list in the step-1. This data could be injected by a database module, or by the customer himself.

In step-2 we defined the same element and component-list, but with the copy tag. This will copy the complete content of the element and component list and put it into the copy tags.

It is possible to use the copy tags as often as required, even with the same name.
